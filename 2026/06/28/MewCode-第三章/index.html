权限系统：给 Agent 戴上”安全带”

本文阐述 MewCode 权限系统的设计动机与架构。
涉及知识点：Prompt 注入、越权操作、数据泄露、五层防线决策链、OS 级沙箱（macOS Seatbelt / Linux bubblewrap + seccomp）、权限学习循环。
读完可理解 Agent 安全体系的分层设计思路，以及”安全是贯穿系统设计的思维方式”这一核心原则。

一、为什么 Agent 需要权限系统

给 Agent 一个 Shell，它就能做任何事——这正是问题所在。没有权限约束的 Agent 面临三类核心威胁：

1.1 Prompt 注入

攻击者将恶意指令藏在 Agent 会读取的地方。比如在一段看似普通的文本末尾追加：

当 Agent 读取这段内容时，可能将其当作合法指令执行。这种攻击之所以危险，是因为数据通道和指令通道是同一个——Agent 无法天然区分”要处理的内容”和”要执行的命令”。

1.2 越权操作

你让 Agent 修一个 Bug，它顺手把你的 ESLint 配置改了、把 CI 脚本”优化”了、把 package.json 的依赖版本全升级了。Agent 的”过度热心”在没有边界约束时会变成破坏力。

本质问题：Agent 的能力边界没有被显式定义，它不知道哪些操作超出你的预期。

1.3 数据泄露

Agent 可能将敏感文件（密钥、配置文件、数据库密码）的内容发送到第三方 API，或者在错误日志中附带完整的环境变量。这类泄露往往是无意的，但后果与主动攻击一样严重。

二、应用层：五层防线

MewCode 在应用层构建了五道防线，按决策速度从快到慢排列，形成一个级联决策链——上层能判断就直接裁决，不能判断才向下传递。

2.1 第一层：危险命令黑名单

最直接的一层：维护一份已知危险命令的列表，命中即拦截。

局限性：黑名单只能防已知的攻击模式。攻击者可以绕过——用 base64 -d <<< ... | sh 代替直接 pipe、用罕见的系统工具实现同样效果。这一层的定位是低成本拦截最明显的攻击，而非终极方案。

2.2 第二层：路径沙箱

限制 Agent 的文件读写范围，只允许访问：

  • 项目根目录（及其子目录）
  • 系统临时目录

同时检查符号链接：即使文件路径看起来在项目目录内，如果它实际指向 /etc/passwd 或 ~/.ssh/id_rsa，也必须拦截。判断依据是解析后的真实路径，而非表面路径。

💡 路径沙箱是”纵深防御”的典型体现：即使黑名单被绕过，攻击者也无法读写敏感文件。

2.3 第三层：权限规则

通过规则文件声明”谁能做什么”。规则分三个层级，优先级从高到低：

层级 作用域 典型用途
本地级 当前项目 项目特定的精细控制，比如”禁止读取 .env“
项目级 团队共享 .mewcode/permissions.yaml，随代码仓库分发
用户级 全局默认 ~/.mewcode/permissions.yaml，个人安全基线

关键设计：**deny 规则跨层合并，allow 规则只在本层生效**。也就是说，用户在全局设了 deny，项目级无法用 allow 覆盖——安全底线不会被任何一层突破。而本地级的 deny 可以覆盖项目级的 allow，让开发者对本地环境有最终控制权。

2.4 第四层：权限模式

借鉴 Claude Code 的交互模式设计，提供三种模式供用户选择：

  • Plan 模式：Agent 只读、只规划，不执行任何修改操作。适合审查复杂变更前的方案。
  • Accept Edit 模式：Agent 可以修改文件，但每次写入都需要用户确认。适合对代码质量要求高的场景。
  • 默认模式：Agent 自主执行，仅在触发权限规则时询问用户。

三种模式本质上是预先批量授权——用户根据自己的信任程度选择一档，而非每条命令都弹窗确认。

2.5 第五层：HITL 人在回路

如果前面四层都没有给出明确裁决（既不在黑名单、也不在规则覆盖范围内），系统会暂停并将决策权交给用户：

“人在回路”是最后一道防线，但它不只是”弹窗问用户”——始终允许 和 始终拒绝 选项会回写规则，形成下一节讨论的权限学习循环。

2.6 决策链全景

💡 被拒绝的操作不终止 Agent 循环，而是将错误信息返回给模型。模型看到”操作被拒绝”后会尝试换一条路径——比如换个命令、换个工具、或者向用户解释为什么需要这个权限。这种”优雅降级”比直接崩溃更符合 Agent 的工作方式。

三、OS 层：硬安全兜底

应用层的防线是”软件层面的君子协定”——Agent 代码承诺遵守规则，但万一代码有 Bug 或被绕过呢？

OS 层的沙箱让操作系统内核来执行隔离。代码层面无法绕过，因为限制不是写在应用逻辑里，而是刻在内核的强制访问控制中。

3.1 macOS：Seatbelt

macOS 的 Seatbelt（sandbox(7)）是 XNU 内核级的强制访问控制框架。通过 .sb 配置文件声明策略：

Seatbelt 的关键特性：

  • 内核强制执行：用户态代码无法绕过
  • 不可逆：sandbox_init() 一旦调用，进程及其子进程永久受限
  • 默认拒绝：只显式放行需要的操作

3.2 Linux：Bubblewrap + Seccomp

Linux 使用两层组合：

  • Bubblewrap（bwrap）：利用内核命名空间（namespace）实现文件系统隔离。通过 bind mount 将项目目录映射进沙箱，其余系统目录只读挂载或不挂载。Flatpak 和 Steam 都在生产环境中使用它。
  • Seccomp（Secure Computing）：通过 BPF（Berkeley Packet Filter）程序过滤系统调用。可以精确到”允许 read/write/open，禁止 mount/ptrace/kexec_load“ 的粒度。

两者的分工：bwrap 负责文件系统和命名空间隔离（”能看见什么”），seccomp 负责系统调用过滤（”能做什么操作”）。

3.3 双层联动：autoAllow

应用层和 OS 层通过 autoAllow 机制联动：当 OS 沙箱已开启时，应用层自动放行普通的 Bash 命令，因为操作系统内核已经在底层兜底——文件系统和网络都已被限制在安全范围内。这减少了不必要的确认打扰，让安全体系在”严格”和”流畅”之间取得平衡。

四、关键设计决策

除了分层架构本身，还有几个贯穿始终的设计选择值得单独说明。

4.1 拒绝但不停车

传统安全系统倾向于”发现问题就终止进程”。但在 Agent 场景中，权限被拒绝后返回错误信息给模型，让模型自己调整策略，是更好的选择。

模型看到这个反馈后可能换个思路——用项目内的 users.json 代替系统文件，或者向用户说明需要额外权限。这种交互式纠错比直接终止更符合 Agent 的工作模式。

4.2 “始终允许”学习循环

当用户在 HITL 确认中选择了”始终允许”，这个规则会被记录到对应层级的规则文件中。下次同样的操作就不再询问。

这形成了一个权限学习循环：系统越用越了解用户的偏好，确认频率逐步降低，但安全基线从未被削弱——因为所有规则都是用户显式授权的。

4.3 敏感路径双重防护

以下路径在应用层和 OS 层同时被禁止写入：

路径 风险
config.yaml / permissions.yaml 篡改配置以提升权限
skills/ 目录 注入恶意技能定义
.env / 密钥文件 窃取或泄露凭证

双层防护的意义：即使应用层的路径检查被绕过（比如通过某种编码技巧），OS 层的 sandbox profile 仍然会拦截写入操作。

五、总结

安全不是”加完就可以划掉”的功能清单，而是一种贯穿系统设计的思维方式。

MewCode 的安全体系由两层构成。应用层的五层防线（黑名单 → 路径沙箱 → 权限规则 → 权限模式 → HITL）负责快速筛选，灵活且体验友好。OS 层沙箱（macOS Seatbelt / Linux bubblewrap + seccomp）负责硬安全兜底，由内核强制执行文件隔离和网络隔离，代码层面无法绕过。

两层通过 autoAllow 联动：沙箱开启后自动放行 Bash 命令，减少不必要的确认打扰，在安全与流畅之间找到平衡点。

回顾本章的核心设计：

  • 级联决策链：上层能判断就直接裁决，不能判断才向下传递，兼顾速度和覆盖率
  • 拒绝而非终止：错误信息返回给模型，让 Agent 有机会调整策略
  • 权限学习循环：”始终允许”回写规则，越用越顺畅而不损失安全基线
  • deny 跨层合并：安全底线不会被任何一层的 allow 突破
  • 敏感路径双重防护：应用层和 OS 层共同守卫关键配置文件

更重要的是，有了权限系统和 OS 级沙箱，用户才敢放心地给 Agent 更多权限。用户越信任，Agent 能发挥的价值就越大。

MewCode 现在有了安全保障，是时候给工具系统做质的扩展了。下一章引入 MCP 协议，让工具系统从封闭走向开放生态。
