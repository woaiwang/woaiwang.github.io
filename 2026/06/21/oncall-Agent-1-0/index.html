oncall Agent 1.0 — RAG 问答系统核心流程与组件拆解

二、核心概念速查表

先搞清楚每个东西是干什么的，后面看代码时才不会迷路。

2.1 组件一览

组件 一句话定位 在这个项目里具体干什么
Docker 容器化部署工具 一键启动 Milvus 服务（包含 etcd、MinIO 等依赖），不用手动装一堆东西，保证开发/生产环境一致
Milvus 专门的向量数据库 存文本对应的语义向量 + 提供毫秒级向量相似度搜索，是 RAG 检索环节的核心存储引擎
LangChain LLM 应用开发框架 串联整个 RAG 流程的骨架——封装了文档加载、文本分割、向量化、检索的通用接口，不用从零造轮子
嵌入模型（Embedding Model） 文本 → 向量的转换器 本项目用阿里云 DashScope 的 text-embedding-v4，把任意文本映射成 1024 维的浮点数向量；语义越接近的文本，向量之间的”距离”越小
本地文件（txt/md） 私有知识库 RAG 的知识来源，存在 ./uploads 目录下

2.2 关键术语

术语 含义 为什么重要
向量（Vector/Embedding） 文本的数学表示，如 [0.12, -0.34, 0.78, ...]（1024个浮点数） 让机器能”算”出两段文本的语义相似度——相似度 = 向量之间的距离
分片（Chunk） 长文本被切成的一个个小段 嵌入模型有最大输入长度限制；分片太小丢失上下文，太大检索精度下降——需要平衡
元数据（Metadata） 附在每个分片上的额外信息，如来源文件路径 用来按文件删除旧数据（delete_by_source），保证索引更新后不残留旧向量
Top-K 检索返回的最相关的前 K 个结果 K 太小可能漏掉答案，K 太大引入噪声——需要根据场景调参
相似度搜索 在向量空间中找出与查询向量”距离最近”的向量 这是 RAG 检索的核心操作——用向量距离代替关键词匹配

三、从磁盘文件到 Milvus 存储：索引全流程详解

这是整个系统最基础也最重要的流程。一句话概括：把磁盘上的非结构化文本文件，转换成可以快速语义检索的向量数据。

3.1 第一步：读取磁盘文件

VectorIndexService.index_single_file() 读取 ./uploads 目录下的 txt/md 文件，拿到原始文本内容。

3.2 第二步：删除旧数据 — 为什么这一步在切片之前？

做法：通过 vector_store_manager.delete_by_source() 按文件路径（存在 metadata 的 _source 字段里）删除该文件之前的所有向量数据，然后再写入新数据。先删后写，保证数据永远是最新的。

3.3 第三步：文档分片 — 长文本 → 小块

不能把整本书直接转成一个向量：嵌入模型有最大输入 token 数限制，而且太长会导致语义信息被”稀释”，检索精度大幅下降。

本项目使用 三阶段优化分割策略（DocumentSplitterService）：

🔑 chunk_overlap=100 是干什么的？ 相邻分片之间有 100 个字符的重叠区。假设关键信息刚好落在分片边界上——“……保研条件如下：（分片结束）1. GPA ≥ 3.5……”——如果没有 overlap，前半句在一个分片，后半句在另一个分片，检索时可能两个都匹配不好。overlap 用很小的成本解决了这个问题。

每个分片被包装成 LangChain 的 Document 对象，metadata["_source"] 记录来源文件路径，方便后续按文件删除。

3.4 第四步：生成文本向量（Embedding）

调用 vector_store_manager.add_documents() 时，LangChain 自动调用 VectorEmbeddingService：

🔑 什么是嵌入（Embedding）？ 把文本映射到高维向量空间中的一个点。映射的质量取决于嵌入模型——好的模型能让”保研”和”推免”的向量距离很近，但它们和”食堂”的向量距离很远。

3.5 第五步：存储到 Milvus

最终，每个分片的以下信息被一起存入 Milvus 的 biz 集合：

存入的内容 说明
向量（1024维 float 数组） 用于相似度搜索
原始文本 检索命中后返回给用户看的
元数据（文件路径等） 用于按来源过滤和删除

🔑 “集合（Collection）”是什么？ 你可以把它类比为关系数据库中的”表”。biz 是这个项目的业务集合名。一个 Collection 里存了所有文件的向量数据，通过 metadata 中的 _source 字段区分来源——这样跨文件检索时不需要跨表查询。

四、代码文件逐个拆解

理解了每个文件是干什么的、为什么这样设计，下次看代码就不会迷路。

4.1 文件上传入口 — app/api/file.py

  • 接收前端传来的文件
  • 验证：只允许 txt/md 格式，大小限制默认 10MB
  • 保存到本地 ./uploads 目录
  • 保存成功后自动触发单个文件的索引流程（调用 VectorIndexService）
  • 返回上传结果（成功/失败 + 错误信息）

💡 为什么上传完自动触发索引？ 减少用户操作步骤。用户只需上传文件，不用再点”开始索引”。对运维场景来说，这种”上传即索引”的体验比手动触发好很多。

4.2 索引流程编排 — app/services/vector_index_service.py

核心逻辑是一个”编排器”（Orchestrator），自己不干活，只负责按顺序调用其他服务：

💡 为什么需要一个专门的编排层？ 把”调度逻辑”和”执行逻辑”分开。如果以后想加一个步骤（比如索引完成后的通知），只需改这一个文件，不用动其他服务。

4.3 文档分片 — app/services/document_splitter_service.py

核心策略就是上面说的三阶段分割（结构切 → 大小切 → 合并碎），这里补充几个设计细节：

  • 为什么先按标题切？ markdown 的标题天然是语义边界，按标题切能尽可能保证每个 chunk 是一个完整的”小节”，而不是在句子中间硬切。
  • chunk_size=800 怎么定的？ 经验值。text-embedding-v4 最大支持约 2048 tokens，800 字符（约 400-500 中文 tokens）既在安全区内，又不会太小导致信息碎片化。
  • 每个分片携带 metadata：_source（文件路径）是必带的，用于后续按文件删除和来源引用。

4.4 向量生成 — app/services/vector_embedding_service.py

  • 封装了阿里云 DashScope 的 text-embedding-v4 模型
  • 输出 1024 维浮点数向量
  • 实现了 LangChain 标准的 Embeddings 接口——这意味着上层调用方（vector_store_manager）不需要知道底层用的是哪个嵌入模型，只需要调 embed_documents() / embed_query()

🔑 为什么要实现 LangChain 标准接口？ 这是 LangChain 生态的惯用做法。vector_store_manager 通过 LangChain 的抽象接口调用嵌入服务，将来换成 OpenAI embedding 或本地 BGE 模型时，只需换一个实现类，其他代码一行不改。

4.5 向量存储管理 — app/services/vector_store_manager.py

这是最重要的服务层，提供三个核心能力：

方法 作用 内部做了什么
add_documents(docs) 批量写入向量库 自动调用 embedding 服务把文本转成向量 → 为每个分片生成唯一 ID → 写入 Milvus
delete_by_source(source) 按来源删除 在 Milvus 中查询 metadata["_source"] == source 的所有数据 → 批量删除
similarity_search(query, k) 相似度检索 把查询文本转成向量 → 在 Milvus 中找 Top-K 最相似的 → 返回 Document 列表

4.6 Milvus 底层连接管理 — app/core/milvus_client.py

核心工作：

  1. 连接池管理：维护与 Milvus 的连接，避免每次请求都重新建连

  2. 初始化 Collection：定义 Schema，包含四个核心字段：

    字段 类型 作用
    id VARCHAR（主键） 唯一标识每个分片
    vector FLOAT_VECTOR(1024) 存储 1024 维语义向量
    content VARCHAR 存储原始分片文本
    metadata JSON 存储文件路径等元数据

  3. 创建索引：使用 IVF_FLAT 索引类型，L2 距离度量

    • IVF_FLAT：倒排文件索引。先把向量空间划分成多个”聚类中心”，搜索时只搜最近的几个聚类，而不是全表扫描。大幅加速检索，代价是可能丢失极少量的精度（近似最近邻搜索）。
    • L2 距离：欧几里得距离，即两个向量之间”直线距离”。值越小表示两个向量越接近，语义越相似。

  4. 兼容性修复：修复了 langchain_milvus 和 pymilvus ORM 之间的兼容性问题

六、关键设计决策问答

把笔记里 AI 回复的设计要点提炼成 Q&A 形式，方便快速回顾。

Q1：为什么要”先删后写”而不是”覆盖”？

Milvus 不原生支持”按文件名覆盖”——因为同一个文件可能被切成几十个分片，每个分片是一条独立的向量记录。如果不先删除，旧版本的分片和新版本的分片会并存，检索时可能同时返回新旧内容，产生矛盾。delete_by_source + add_documents 的组合保证了数据更新是原子性的——先全删、再全写。

Q2：为什么选用 IVF_FLAT 索引而不是其他索引？

IVF_FLAT 是 Milvus 默认的索引类型，平衡了速度和精度：

  • FLAT 意味着每个聚类内部的向量是”平铺”的——搜索时在聚类内部做的是精确距离计算，不压缩向量
  • IVF（倒排文件）意味着搜索时只搜最近的几个聚类，大幅减少计算量
  • 对于本项目的中等规模知识库，IVF_FLAT 足够快（毫秒级）且精度高

相比之下，HNSW 索引在图结构上搜索，速度更快但内存占用更大；PQ 索引压缩向量但精度损失较多。IVF_FLAT 是”够用且稳妥”的选择。

Q3：chunk_overlap=100 和 chunk_size=800 是什么关系？

如果一个文档被切出 3 个分片：

100 个字符在中文里大约 2-3 句话。这个冗余量成本几乎为零（只多了 ~12.5% 的存储），但能有效防止关键信息落在分片边界上被”拦腰截断”。

Q4：LangChain 在这套系统里到底扮演什么角色？

LangChain 是胶水框架，它不提供任何 AI 能力本身，而是提供了统一的接口抽象：

  • Document：统一的数据结构，所有分片都用它表示
  • Embeddings：统一的嵌入接口，切换模型只换实现
  • VectorStore：统一的向量存储接口，切换向量数据库只换实现
  • TextSplitter：统一的分片接口

简单说：LangChain 让你的代码不依赖具体实现。今天用 DashScope embedding + Milvus，明天想换成 OpenAI embedding + ChromaDB，只需要换两个实现类，业务逻辑（vector_index_service.py）一行不用改。

Q5：为什么 metadata 里要存 _source？

两个作用：

  1. 按文件删除旧数据：delete_by_source 靠的就是这个字段来找到”哪些向量属于这个文件”
  2. 检索结果溯源：用户看到回答后可以看到每条引用来自哪个文件，增强可信度

七、一句话总结

你要记住的 一句话
整个系统在干什么 把本地知识库文件（txt/md）向量化存入 Milvus，用户提问时从 Milvus 检索相关内容作为上下文，让大模型基于私有知识准确回答
核心数据流 文件 → 分段 → 向量化 → Milvus → 查询 → 检索 → 拼接 Prompt → 大模型回答
6 个核心文件各司何职 file.py 收文件 → index_service.py 编排流程 → splitter_service.py 切文本 → embedding_service.py 转向量 → store_manager.py 管存取 → milvus_client.py 管连接
最关键的工程细节 先删后写保证数据一致性、三阶段分片保留语义结构、LangChain 接口解耦具体实现、metadata 实现按文件管理
