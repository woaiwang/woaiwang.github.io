一、全景速览

从 1.0 到 2.0，Oncall Agent 经历了两个核心变化：

一句话概括 2.0：Oncall Agent 2.0 是一个基于 ReAct 推理范式 和 Plan-Execute-Replan 多智能体架构 的智能运维助手，通过 MCP 协议 集成腾讯云 CLS 等外部工具，实现对告警和日志的智能分析。

二、ReAct 架构：推理 + 执行 + 观察的闭环

2.1 什么是 ReAct？

ReAct = Reasoning + Acting（推理 + 行动），是 LLM Agent 领域最经典的架构范式。

它的核心思想极其简单：让大模型在一个循环里不断地”思考 → 行动 → 观察结果 → 再思考”，直到解决问题。

🔑 为什么叫 ReAct？ 因为它的每一步都在交替做两件事：Reason（推理）和 Act（行动）——推理告诉你”该做什么”，行动去”实际做”，观察结果反馈给下一轮推理。这就是一个完整的 感知-决策-执行 闭环。

2.2 古老 ReAct：纯文本驱动的”协议”

在 Function Call 出现之前，ReAct 是这样实现的：

做法：在 System Prompt 里写死一套文本格式规范，强制大模型按照规定的文本模式输出。

大模型收到后按格式输出，应用程序通过正则表达式解析这段文本，提取 Action 和参数，执行工具，把结果拼回去，再让大模型继续。

古老 ReAct 的致命缺陷：

问题 具体表现
格式脆弱 大模型可能输出 Thoughts:（多了一个 s）、Action input:（大小写不一致），正则匹配直接失败
参数复杂时易出错 嵌套 JSON 参数容易格式错误、少括号、多引号，解析失败率随参数复杂度指数上升
幻觉行动名 大模型可能编造一个不存在的工具名，程序调用失败后才报错
调试困难 出问题时，你在看一段自由文本而非结构化数据，定位问题靠肉眼
不支持流式 文本格式解析必须等待完整输出，无法边生成边解析

💡 打个比方：古老 ReAct 就像你给一个人发短信说”你要严格按这个格式回复我”，但对方可能手滑、打错字、格式跑偏——你没有任何机制来保证他一定遵守格式。

2.3 现代 ReAct：Function Call（函数调用）—— 从文本到 JSON

Function Call 是现代 LLM（GPT-4、Claude 等）原生支持的能力。它的核心改进是：不再用文本约定，而是用结构化的 JSON Schema 定义工具，大模型以 JSON 格式返回调用请求。

🔑 关键差异：古老 ReAct 是大模型”写作文”给程序看，程序靠正则”猜”他写了什么；Function Call 是大模型直接输出结构化的 JSON，程序不需要”猜”——格式由协议保证，出错概率趋近于零。

2.4 ReAct 架构的优势总结

优势 说明
架构简单 只需要一个推理-执行-观察的循环，就能处理各种复杂任务
可解释性强 每一步的 Thought 和 Action 都有记录，可以完整回溯推理链
输出结构化 使用 Function Call 后，工具调用是标准 JSON，解析零歧义
易于扩展 新增工具只需在 tools 数组中加一个 JSON Schema，不影响现有逻辑
错误可见 工具调用失败时，错误信息会被喂回大模型，让它自我修正

三、Plan-Execute-Replan 架构：多智能体协作模式

3.1 核心思想

Plan-Execute-Replan 是 ReAct 的升级版——把”一个人边想边做”升级为”一个人先想好计划、另一个人去执行、第一个人再检查要不要调整计划”的 多角色协作模式。

🔑 类比理解：ReAct 像一个工程师接到需求就边想边干；Plan-Execute-Replan 像一个团队——项目经理先拆任务（Planner）、工程师执行（Executor）、项目经理根据进度调整计划（Replan）。

3.4 Plan-Execute-Replan 架构的优势

优势 说明
全局规划能力强 Planner 在动手前先全局思考，避免”见树不见林”——ReAct 容易在局部细节中迷失
容错性和自愈性 Replan 角色能在某步骤失败后自动调整策略，而非直接报错退出
职责分离清晰 战略层(Planner)和执行层(Executor)解耦，各自优化各自的能力
可并行执行 无依赖的子任务可以被多个 Executor 并发执行，提升效率
更长的任务链 有全局计划支撑，可以处理比纯 ReAct 更长、更复杂的多步骤任务

四、ReAct vs Plan-Execute-Replan：多维对比

从以下维度对比两种架构，帮助你在不同场景下做出正确的技术选型。

对比维度 ReAct Plan-Execute-Replan
核心思路 边想边做，推理和行动交替进行，无全局计划 先规划后执行，有显式的全局计划，执行中可调整
是否有全局计划 ❌ 没有。每一步只看到当前的观察结果 ✅ 有。Planner 先制定完整计划再分发
Agent 数量 1 个（单一 Agent 循环） 多个（至少 3 个角色分工：Planner / Executor / Replan）
任务进度感知 弱。Agent 可能在中途”忘记”最初的复杂目标 强。Planner 持有任务清单，可追踪每个子任务的完成状态
应对变化能力 有限。遇到意外结果只能在下一步调整，没有全局视角 强。Replan 角色专门负责根据执行反馈调整全局计划
适合场景 简单到中等复杂度的任务（单次查询、信息检索、简单推理） 复杂的多步骤任务（故障排查、多源数据分析、长流程自动化）
适合的任务长度 短到中等（1~5 个步骤） 中到长（5~20+ 个步骤）
错误恢复 靠大模型自身判断”这次行动失败了，换个方式” 有专门的 Replan 角色审视失败原因并系统性调整
并发能力 无（单线程循环） 有。无依赖的子任务可被多个 Executor 并行处理
调试透明性 中等。可以看 Thought → Action → Observation 链路 高。Plan → 每个 Executor 的结果 → Replan 决策 → 最终 Plan，分阶段可追踪
实现复杂度 低。一个循环 + Function Call 即可 中到高。需要三个角色的 Prompt 工程、状态管理、通信协议
延迟 低。无额外规划开销 中。Planner 规划 + Replan 审查有额外 LLM 调用成本

架构选择决策树

💡 Oncall Agent 2.0 的选择：运维排障天然是多步骤、跨数据源的复杂任务（查告警 → 查日志 → 查指标 → 综合分析），所以 2.0 采用 Plan-Execute-Replan 架构是正确的选择。

六、MCP 协议深度解析

MCP（Model Context Protocol，模型上下文协议） 是 Anthropic 于 2024 年 11 月推出的开放协议，被比喻为 **”AI 应用的 USB-C 接口”**。理解 MCP 是理解 Oncall Agent 2.0 工具集成方式的关键。

6.1 MCP 解决了什么问题？

传统 AI 应用集成外部工具的痛点（M × N 问题）：

🔑 一句话理解 MCP：MCP 定义了一套标准协议，让任何 AI 应用（Client）都能以同一种方式连接任何工具/数据源（Server），不需要为每个组合写专用代码。就像 USB-C 一样——一个接口，连接所有外设。

6.2 MCP 架构：Host-Client-Server 三层模型

三层角色的职责

层级 角色 核心职责 类比
Host 主机 运行 AI 应用的容器，管理所有 Client 的生命周期、权限控制、用户交互 操作系统
Client 客户端 与一个 Server 建立 1:1 连接，双向路由协议消息，维护会话状态和安全隔离 USB 控制器
Server 服务端 暴露 Resources / Tools / Prompts 三大能力，独立运行，专注单一职责 USB 设备

Server 暴露的三大核心能力

6.3 通信机制：JSON-RPC 2.0

MCP 的所有消息都通过 JSON-RPC 2.0 格式传输，定义了三种消息类型：

消息类型 方向 说明 示例
Request 双向 带 id 的消息，期待一个 Response 调用工具、读取资源
Response 双向 对某个 Request 的成功/错误响应 工具返回值、资源内容
Notification 双向 无 id 的消息，不需要回复 资源变更通知、状态更新

两种传输方式：

方式 适用场景 通信机制
STDIO 本地进程 通过标准输入/输出流传输 JSON-RPC 消息，简单高效
Streamable HTTP 远程服务 统一 /message 端点，支持 SSE 流式推送，适合团队共享和远程部署

6.5 动态发现机制 — MCP 最强大的特性

动态发现 是 MCP 协议的核心创新：AI 应用不需要在代码中硬编码”有哪些工具”，而是在运行时通过协议动态获取。

6.6 MCP 的安全设计

安全原则 具体实现
用户同意 所有数据访问和工具调用需用户明确授权
数据隐私 Host 必须获得用户同意才能向 Server 暴露数据
服务器隔离 每个 Client-Server 连接独立，Server 无法读取其他 Server 的数据，也无法读取完整对话历史
最小权限 生产环境使用子账号，只授予必要的权限（如 CLS 只读权限 QcloudCLSReadOnlyAccess）

7.7 为什么用 MCP 而不是直接调 API？

对比维度 直接调 CLS API 通过 CLS MCP Server

八、关键设计决策问答

Q1：为什么不只用一个 ReAct Agent，而要引入 Plan-Execute-Replan？

运维排障场景的特点是 步骤多、跨数据源、执行路径不确定（你可能查到一半才发现需要查另一种日志）。纯 ReAct 容易在 3-4 步之后”忘记”最初的目标，Plan-Execute-Replan 的 Planner 持有一份显式的任务清单，能保证不遗漏。而且 Replan 角色能在”执行结果为空”时自动调整策略，而非直接说”我找不到”。

Q2：CLS MCP 的 TextToSearchLogQuery 是怎么做到的？

它内部调用大模型（或 CLS 自己的 NL2SQL 引擎）将自然语言翻译为 CQL（Cloud Query Language）——腾讯云日志服务的查询语言。这一步本质上是一次”语义翻译”：把”过去一小时的错误日志”翻译成 level:ERROR AND timestamp:[now-1h TO now]。翻译结果会作为 SearchLog 的参数执行。

Q3：MCP 和传统的 REST API 有什么本质区别？

REST API MCP
发现机制 需要看文档才知道有哪些接口 tools/list 自动获取，LLM 自行理解
调用方式 开发者写代码调用 LLM 自主决定何时调用、传什么参数
扩展性 新增 API → 改代码 → 部署 Server 声明新工具 → 自动通知 → 即时可用
标准化 每个 API 格式不同 统一的 JSON-RPC 2.0 + JSON Schema

MCP 是为 AI Agent 自动调用工具 设计的，REST API 是为 人类开发者写代码调用 设计的。

Q4：CLS MCP Server 的安全性如何保证？

  • 密钥管理：通过环境变量 TENCENTCLOUD_SECRET_ID / TENCENTCLOUD_SECRET_KEY 传入，不硬编码
  • 最小权限：建议使用子账号，只授予 QcloudCLSReadOnlyAccess（只读权限），防止 Agent 误删日志
  • 内网安全：如在腾讯云 VPC 内网，API 域名设为 internal.tencentcloudapi.com，流量不出公网
  • MCP 协议层：每次工具调用需经过 MCP Host 的权限检查，用户可拦截敏感操作

Q5：Oncall Agent 2.0 的 Planner 和 Executor 之间是怎么通信的？

Planner 不直接和 Executor 通信——它们都通过一个统一的 消息总线 / 状态管理器 交换信息：

Planner 把计划写入任务队列，Executor 从队列取任务执行，结果回写到状态管理器，Replan 读取所有结果后决定是否调整计划。这种松耦合设计让每个角色的实现可以独立迭代。

十、一句话总结

你要记住的 一句话
ReAct 是什么 推理→行动→观察→再推理的循环——大模型边想边做，每一步都是 Thought → Action → Observation
古老 vs 现代 ReAct 古老 ReAct 靠正则解析文本（脆弱），现代 ReAct 用 Function Call 输出标准 JSON（可靠）
Plan-Execute-Replan 把一个人循环升级为三个角色协作：Planner 拆任务 → Executor 执行 → Replan 调整计划

MCP 协议核心价值 AI 的”USB-C”——标准协议让 AI 应用动态发现和调用任何工具，告别 M×N 集成噩梦
CLS MCP Server 把腾讯云日志服务封装成 MCP 标准工具，让 AI 能用自然语言直接查日志、看指标、管告警
