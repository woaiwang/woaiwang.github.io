二、踩坑记录 🐞 —— 最重要的一节

坑：VS Code 死活连不上远程服务器

现象：

在 VS Code 中配置好 SSH Host 后，反复尝试连接，始终失败。错误信息大致为连接超时或无法建立 SSH 通道。

排查过程：

检查了以下项，全部正常：

  • ✅ 服务器 IP 和端口正确
  • ✅ 用户名和密码正确
  • ✅ 服务器 sshd 服务在运行
  • ✅ 本地网络正常

但仍然连不上。😫

真正的原因（根因分析）：

SSH 连接走了本地代理，而代理本身不可用或指向了错误的端口。

具体来说：我之前在 VS Code 的用户设置（Preferences: Open User Settings (JSON)）中配置过 HTTP/HTTPS 代理（可能用于科学上网或其他网络调试），这个代理设置也被 VS Code 的 Remote-SSH 插件一并继承了。Remote-SSH 在建立 SSH 连接时，默认会使用系统或 VS Code 中配置的代理设置，导致 SSH 流量被错误地发往代理服务器，而不是直接连接目标服务器。

一句话总结根因： 代理配置污染了 SSH 连接通道。SSH 需要直连服务器，但代理设置让它”绕了远路”，结果绕不通。

层次 说明
表象 VS Code 连不上远程服务器
直接原因 SSH 流量走了代理
根本原因 User Settings JSON 中残留了之前配置的代理端口，Remote-SSH 插件的流量被这个代理劫持了

解决方法：

打开 VS Code → Ctrl+Shift+P → 搜索 Preferences: Open User Settings (JSON) → 找到类似以下的代理配置项 → 删掉它：

删除后保存，重新连接 SSH → 秒连 ✅

以后怎么避免：

  1. 代理配置只在需要时开启，用完就关。 或者在配置代理时明确指定作用范围，不要让全局代理影响 SSH。
  2. 连接 SSH 之前，养成检查 VS Code 的代理设置的习惯。 路径：File → Preferences → Settings → 搜索 "proxy"。
  3. 把解决方法记在心里： 凡是 “网络连接类” 问题，第一反应就是排查代理设置。这个排查方向适用于 VS Code、终端、浏览器等几乎所有工具。
  4. 使用 ssh -v 命令调试： 在终端中手动 ssh -v user@host，可以看到详细的 SSH 连接日志，如果走了代理，日志里会有明确显示。

三、今日学到的知识点 📚

3.1 Linux 与 Ubuntu 的关系

  • Linux = 内核（发动机），负责管理硬件、进程、内存、网络
  • Ubuntu = 发行版（整车），在内核基础上加上图形界面、包管理器（apt）、桌面环境等

为什么 AI 实训用 Ubuntu 而不是 Windows？

  • 🎯 CUDA 兼容性：N 卡 + CUDA + PyTorch 在 Ubuntu 上是”原配”，Windows 上配置容易出各种奇怪问题
  • 🖥️ 服务器生态：绝大多数 AI 服务器跑的都是 Linux，学 Ubuntu 就是学真实工作环境
  • ⚡ 命令行效率：批量操作、脚本自动化、依赖管理——全在终端里搞定，比 GUI 快得多

3.2 VS Code Remote-SSH 的工作原理

  • VS Code 的界面跑在本地
  • 文件操作、命令执行、代码运行都发生在远程
  • Remote-SSH 插件在中间建立了一条加密隧道，让你”感觉”像在操作本地文件

关键判断： 在远程终端里 mkdir 创建的文件夹，不会出现在本地桌面上——它在远程服务器的硬盘里。

3.3 四层环境层级结构

这是今天最核心的知识框架，请牢记：

两个”不等于”：

  • apt 管理的是 Layer 1（系统级软件），pip 管理的是 Layer 4（Python 库），不要混用
  • pip install 安装的包不一定属于整个系统——它属于当前激活的那个 Conda 环境

3.5 source ~/.bashrc 是干什么的？

简单类比：~/.bashrc 是终端启动时”自动执行的初始化脚本”。当你用 conda init 修改了这个文件后，当前已打开的终端不会自动加载，必须手动 source ~/.bashrc 让它重新读一遍，这样 conda 命令才会在当前终端中生效。

六、核心原则总结（下次不要再犯）

🚫 错误 1：代理设置导致 SSH 连不上

根因： 全局代理配置污染了 SSH 连接

以后这么做：

  1. 遇到网络连接问题 → 第一步就查代理设置
  2. 代理用完就关，不要长期挂着
  3. 终端里用 ssh -v user@host 调试，看详细日志

🚫 错误 2：分不清 apt 和 pip

根因： 没有理解系统层和 Python 层的区别

以后这么做：

  • 装系统工具（tree、git、vim）→ 用 apt
  • 装 Python 库（torch、numpy、pandas）→ 用 pip（在正确的 Conda 环境下）

🚫 错误 3：把所有东西都装进 base 环境

根因： 不理解环境隔离的意义

以后这么做：

  • base 环境 = 纯净，不动它
  • 每个项目 = 一个独立的 Conda 环境
  • 进入项目前先 conda activate <环境名>
  • 不确定在哪个环境？→ which python 看路径

🚫 错误 4：随意使用 rm 命令

根因： 不知道 Linux 下删除是永久性的

以后这么做：

  • rm 前先 ls 确认文件存在
  • 删除前先用 cat 看一眼内容
  • 重要的文件先 cp 备份再操作
  • 养成好习惯：先 ls → 确认 → 再 rm
